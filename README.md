**Автотесты для проекта "Реальный кейс от компании «Ростелеком Информационные Технологии»".**

Основная цель тестирования — проверить работоспособность функций сайта Ростелеком.

Объект тестирования: система авторизации https://lk.rt.ru/.

Задачи тестирования:
1. Проверка соответствия требованиям.
2. Обнаружение и идентификация дефектов

Методология тестирования:
Автоматизированное тестирование — автотесты на Python + Selenium для процесса авторизации.

Инструменты тестирования:
1. Pytest — запуск тестов. Позволяет создавать тесты в виде функций и классов, а также 
предоставляет удобные методы для проверки условий (assert). Что проверялось: Функциональности веб-приложения, 
включая авторизацию, регистрацию и восстановление пароля. Что сделано: Созданы тесты для тестирования различных сценариев.
Использованы методы assert для проверки ожидаемых результатов.
2. Selenium — автоматизация пользовательских действий в браузере. Позволяет имитировать действия пользователя, такие как ввод
текста, нажатие кнопок и ожидание загрузки страниц. Что проверялось: Функциональность веб-приложения, включая загрузку страниц,
наличие элементов на странице, авторизацию. Что сделано: Использованы методы Selenium для взаимодействия с элементами веб-страниц,
такие как find_element, send_keys, click.
3. time.sleep — используется для ожидания загрузки страниц или выполнения действий. Что сделано: Добавлены задержки для ожидания
ввода капчи и загрузки страниц, чтобы дождаться появления нужных элементов.
4. Техники тест-дизайна — позволяют оптимизировать процесс тестирования - сократить количество тест-кейсов без потери качества:
- Позитивное/негативное тестирование - более эффективное предугадываниие ошибок. Что сделано:14 позитивных и 7 негативных теста.
- Граничные значения - сокращение тестовых случаев. Что сделано: использовано в тестировании ввода телефона и лицевого счета.
5. Excel — позволяет структурировать и документировать тестовые сценарии. Что сделано: Созданы тест-кейсы и баг-репорты для каждого
сценария тестирования, включая шаги, ожидаемые результаты и фактические результаты.

Позитивные тестовые сценарии:
1. Загрузка страницы "Вход по временному коду". Проверить, что пользователь успешно загрузит страницу "Вход по временному коду"
2. Поле "E-mail или мобильный телефон" отображается. Проверить, что на странице "Вход по временному коду" отображается поле "E-mail или мобильный телефон".
3. Кнопка "Получить код" отображается.	Проверить, что на странице "Вход по временному коду" отображается кнопка "Получить код".
4. Кнопка "Войти со своим паролем" отображается.	Проверить, что на странице "Вход по временному коду" отображается кнопка "Войти со своим паролем".
5. Заголовок "войти другим способом" отображается.	Проверить, что на странице "Вход по временному коду" отображается заголовок "войти другим способом".
6. Ссылка на пользовательское соглашение отображается.	Проверить, что на странице "Вход по временному коду" отображается ссылка на пользовательское соглашение.
7. Ссылка на раздел "Помощь" отображается.	Проверить, что на странице "Вход по временному коду" отображается ссылка на раздел "Помощь".
8. Кнопка "Получить код" доступна для нажатия с валидным email.	Проверить, что пользователю доступна для нажатия кнопка "Получить код" с введенным валидным email.
9. Кнопка "Получить код" доступна для нажатия с валидным телефоном.	Проверить, что пользователю доступна для нажатия кнопка "Получить код" с введенным валидным телефоном.
10. Ввод валидного email.	Проверить, что пользователь получит код авторизации при вводе валидного email.
11. Ввод валидного телефона 12 цифр.	Проверить, что пользователь получит код авторизации при вводе валидного телефона.
12. Переход на страницу пользовательского соглашения.	Проверить, что пользователь успешно загрузит страницу с текстом пользовательского соглашения.
13. Переход на страницу раздела "Помощь".	Проверить, что пользователь успешно загрузит страницу раздела "Помощь".
14. Загрузка формы входа в ЛК с паролем.	Проверить, что пользователь успешно загрузит форму входа в ЛК с паролем.

Негативные тестовые сценарии:
1. Ввод email некорректной формы.	Проверить, что пользователь не сможет авторизоваться с некорректными email.
2. Ввод email кириллическими буквами.	Проверить, что пользователь не сможет авторизоваться с некорректными email.
3. Ввод невалидного телефона - 10 цифр.	Проверить, что пользователь не сможет авторизоваться с невалидным телефоном.
4. Ввод невалидного телефона - 12 цифр.	Проверить, что пользователь не сможет авторизоваться с невалидным телефоном.
5. Ввод в поле "E-mail или мобильный телефон" пробела.	Проверить, что пользователь не сможет авторизоваться с пустым полем ввода.
6. Ввод невалидного лицевого счета 11 цифр.	Проверить, что пользователь не сможет продолжить авторизацию с невалидным лицевым счетом.
7. Ввод несуществующего лицевого счета 12 цифр.	Проверить, что пользователь не сможет продолжить авторизацию с несуществующим лицевым счетом.

Для запуска проекта:
1. Клонировать репозиторий.
2. Установить зависимости.
pip install -r requirements.txt
3. Запустить файл:
   pytest -v tests/test_auth_page.py
